# Contract: Spotify Auth API Endpoints & Mobile UI

**Feature**: `004-spotify-player-poc` | **Date**: 2026-02-28 (updated 2026-03-01)
**Type**: REST API (Echo backend) + Mobile UI contracts (Flutter)

---

## Backend API Endpoints

These endpoints extend the existing Echo auth API (`/v1/auth/`) to support Spotify OAuth 2.0 PKCE
token exchange. **Unchanged** — both player screens share the same backend.

---

### POST /v1/auth/spotify/token

Exchanges a Spotify authorization code (obtained by the mobile app via PKCE) for Echo-native opaque
access and refresh tokens. The backend performs the Spotify token exchange and stores Spotify
credentials server-side. The mobile app never receives a Spotify token.

#### Request

```
POST /v1/auth/spotify/token
Content-Type: application/json
```

```json
{
  "code": "AQD...",
  "code_verifier": "dBjftJeZ...",
  "redirect_uri": "https://echo.example.com/auth/callback"
}
```

| Field           | Type     | Required | Description                                                                                   |
|-----------------|----------|----------|-----------------------------------------------------------------------------------------------|
| `code`          | `string` | Yes      | Authorization code returned by Spotify in the redirect                                        |
| `code_verifier` | `string` | Yes      | Plaintext PKCE code verifier (43–128 chars); generated by the mobile app                      |
| `redirect_uri`  | `string` | Yes      | The redirect URI used in the original `/authorize` call; must be on the server-side allowlist |

#### Response — 200 OK

```json
{
  "access_token": "echo_at_...",
  "refresh_token": "echo_rt_...",
  "token_type": "bearer",
  "expires_in": 900
}
```

#### Response — Error Cases

| HTTP Status                | Condition                                                  |
|----------------------------|------------------------------------------------------------|
| `400 Bad Request`          | Missing or invalid fields; `redirect_uri` not on allowlist |
| `401 Unauthorized`         | Spotify rejected the code exchange (invalid/expired code)  |
| `422 Unprocessable Entity` | Pydantic validation failure                                |
| `503 Service Unavailable`  | Spotify API unreachable or rate-limited                    |

### POST /v1/auth/spotify/refresh

Refreshes the Echo token pair. The backend proactively refreshes the underlying Spotify access token
if it is within 60 seconds of expiry.

#### Request

```
POST /v1/auth/spotify/refresh
Content-Type: application/json
```

```json
{
  "refresh_token": "echo_rt_..."
}
```

#### Response — 200 OK

```json
{
  "access_token": "echo_at_...",
  "refresh_token": "echo_rt_...",
  "token_type": "bearer",
  "expires_in": 900
}
```

#### Response — Error Cases

| HTTP Status               | Condition                                                                                                                              |
|---------------------------|----------------------------------------------------------------------------------------------------------------------------------------|
| `401 Unauthorized`        | Echo refresh token invalid or expired; or Spotify refresh token rejected (user revoked access) — mobile must re-initiate Spotify login |
| `503 Service Unavailable` | Spotify API unreachable during token refresh                                                                                           |

### GET /v1/tracks/{track_id}

Proxies a Spotify Web API track metadata request. Requires a valid Echo access token. The backend
calls `https://api.spotify.com/v1/tracks/{id}` using the stored Spotify access token for the
authenticated user and returns a simplified response. Used by **both** player screens.

#### Request

```
GET /v1/tracks/{track_id}
Authorization: Bearer echo_at_...
```

| Path Param | Type     | Description                |
|------------|----------|----------------------------|
| `track_id` | `string` | Spotify track ID (not URI) |

#### Response — 200 OK

```json
{
  "id": "4iV5W9uYEdYUVa79Axb7Rh",
  "uri": "spotify:track:4iV5W9uYEdYUVa79Axb7Rh",
  "name": "Song Title",
  "artist_name": "Artist Name",
  "album_art_url": "https://i.scdn.co/image/...",
  "duration_ms": 213000
}
```

#### Response — Error Cases

| HTTP Status               | Condition                               |
|---------------------------|-----------------------------------------|
| `401 Unauthorized`        | Missing or invalid Echo access token    |
| `404 Not Found`           | Track not found in Spotify              |
| `503 Service Unavailable` | Spotify API unreachable or rate-limited |

## Mobile UI Contracts

### Login Screen

```
SpotifyLoginScreen
  ├─ "Open Player" button
  │    → Navigator.pushReplacementNamed('/player')
  └─ "Open WebView Player" button
       → Navigator.pushReplacementNamed('/player-webview')
```

Both buttons are only active after a successful Spotify OAuth flow. Both button labels are
externalised in ARB (`openPlayer`, `openWebViewPlayer`).

---

### Native SDK Player Screen (`/player` — unchanged)

```
PlayerScreen(
  controller: PlayerController   // provides TrackPlaybackState stream
)

PlayerController(
  stream: Stream<TrackPlaybackState>   // injected; defaults to SDK stream in production
)
```

**`TrackPlaybackState`** fields exposed to the widget:

- `isPlaying: bool`
- `positionMs: int`
- `lastPositionTimestamp: DateTime`
- `currentTrack: Track?`
- `restrictions: PlaybackRestrictions`

**Commands** (methods on `PlayerController` called by the widget):

- `play()` → `SpotifySdk.resume()`
- `pause()` → `SpotifySdk.pause()`
- `seekTo(int positionMs)` → `SpotifySdk.seekTo(positionMs)`
- `skipNext()` → `SpotifySdk.skipNext()`
- `skipPrevious()` → `SpotifySdk.skipPrevious()`

**Queue availability** (exposed for button disabled states):

- `hasPrevious: bool`
- `hasNext: bool`

### WebView Iframe Player Screen (`/player-webview` — new)

```
PlayerWebViewScreen(
  queueRepository: SpotifyQueueRepository,   // provides Queue with hardcoded URIs
  trackRepository: SpotifyTrackRepository,   // fetches Track metadata via Echo backend
  authService: SpotifyAuthService,           // provides Echo access token for API calls
)
```

**`SpotifyIframeWidget`** interface:

```
SpotifyIframeWidget(
  trackId: String,          // Spotify track ID; constructs embed URL:
                            //   https://open.spotify.com/embed/track/<trackId>
  onLoaded: VoidCallback,   // fired on InAppWebView.onLoadStop
  onError: VoidCallback,    // fired on InAppWebView.onLoadError
)
```

- Uses `AutomaticKeepAliveClientMixin` on its `State` to prevent WebView recreation on rebuilds.
- Navigating to a new track calls `InAppWebViewController.loadUrl()` with the new embed URL.
- The Echo access token is **never** passed into the iframe or the WebView context.
- `InAppWebViewSettings.javaScriptEnabled: true` required for the embed to render.

**`WebViewLimitationBanner`** interface:

```
WebViewLimitationBanner()   // stateless; renders AppLocalizations.of(context).webViewLimitationNotice
```

- Always visible when the screen is in `data` state (FR-025).
- No dismiss action; it is a persistent informational notice.

**`PlayerWebViewScreen` state machine** (Dart-managed; no JS bridge):

| State | Trigger | UI shown |
|-------|---------|----------|
| `loading` | Route opened; metadata fetch or iframe load in progress | `CircularProgressIndicator` |
| `error` | Metadata fetch failed OR `onLoadError` fired | Error message + retry button |
| `data` | Metadata loaded AND `onLoadStop` fired | Iframe + `WebViewLimitationBanner` + prev/next buttons |

**Queue navigation** (Dart-only; no JS bridge):

- Prev/next taps update `Queue.currentIndex` in Dart.
- `SpotifyIframeWidget` receives the updated `trackId` and calls `loadUrl()` to reload the iframe.
- Prev button is disabled when `Queue.hasPrevious == false`.
- Next button is disabled when `Queue.hasNext == false`.

**No JS bridge is used** on the WebView iframe screen. The iframe is an opaque embed; Dart
interacts with it only by loading/reloading URLs. There are no `addJavaScriptHandler` calls and no
`callAsyncJavaScriptFunction` calls on this screen.
